# MicroClaw v0.3.0 — "Self-Improving Runtime"

Status: **proposal** · Target: next brand-new minor (0.2.x → 0.3.0) · Author: planning pass
Baseline: v0.2.2 (0.2.0 milestone = concurrent specialist team, humanlike behavior, graph
memory, multimedia, expanded channels)

This plan synthesizes two inputs:

1. **The maintainer's own quarterly roadmap** (issue #378, "2026 H2–2027 H1 parity targets
   vs openclaw and hermes-agent") and the in-repo `docs/hermes-agent-optimization-plan.md`.
2. **A fresh survey of adjacent projects and 2026 techniques** — nanoclaw (container
   isolation + skills-as-modules), Letta/MemGPT (self-editing memory blocks + sleep-time
   compute), the Claude Agent SDK (skills / subagents / context engineering), the 2026
   agent-sandbox consensus (microVM / gVisor / hardened container defense-in-depth), and
   the 2026 eval/guardrail stack (trajectory evals, boundary guardrails).

The theme deliberately matches issue #378's **2026 Q3 "Self-improving runtime"** quarter so
this becomes the concrete release plan for that quarter, not a parallel track. Visual/Canvas,
mobile nodes, and serverless hibernate (the #378 Q4/2027 items) are intentionally deferred to
v0.4.0 and noted at the end.

---

## Why this theme

MicroClaw already *executes* well: 37 tools, 17 channels, concurrent specialists, graph
memory, MCP, OTLP observability. The next leap is the runtime getting **better the longer it
runs** — curating its own memory, distilling its own skills, spending the right model on the
right sub-task, and doing it inside a sandbox you can actually trust. Every pillar below maps
to a technique that has matured in the ecosystem since 0.2.0 shipped.

| Pillar | Ecosystem signal | MicroClaw today | Gap closed by 0.3.0 |
|---|---|---|---|
| 1. Self-curating memory | Letta self-editing blocks + sleep-time compute | reflector + graph recall, all online | offline consolidation pass, USER.md separation, capacity bounds |
| 2. Skill curator | hermes-agent skill curator; Claude Agent SDK skills | 41 static skills + `skill_manage` | agent distills *new* skills from its own trajectories |
| 3. Per-task auxiliary models | hermes-agent aux models | one model does everything | cheap model for compaction/title/vision/summarize |
| 4. Sandbox pluralization | 2026 microVM/gVisor/hardened-container consensus | single sandbox abstraction | Docker + gVisor + SSH backends, defense-in-depth |
| 5. Evals & guardrails | 2026 trajectory-eval + boundary-guardrail stack | regression packs + OTLP traces | replayable trajectory eval harness + boundary guardrails |

## Competitive landscape & strategic direction (2026 survey)

A second research pass on the two reference projects and the broader field confirmed the
theme and sharpened the *positioning*.

**Where the reference projects are heading**

- **OpenClaw → enterprise multi-agent platform.** v4.0 (mid-2026) targets native multi-agent
  orchestration, a typed-contract plugin SDK, built-in vector memory (ChromaDB), an operator
  dashboard, enterprise integrations (Teams/Salesforce), SSO/SAML + audit logging, a move of
  runtime state into SQLite, and a "Proxyline" egress-routing layer that blocks private/
  metadata destinations. Direction = heavy, enterprise, SaaS-shaped, compliance-driven.
- **Hermes Agent → "grows with you" + RL training infrastructure.** Self-improvement plus a
  trajectory-data factory: parallel tool-calling trajectory generation, checkpointing,
  Atropos RL integration and trajectory export, backed by the Psyche decentralized training
  network. Direction = self-evolution + turning agent behavior into trainable data for the
  next model.

**What the broader field is converging on (2026)**

1. **Self-evolving / lifelong learning is the strongest consensus** — persistent agents that
   improve through *explicit memory and skill growth* (generate → store → reuse), not
   fine-tuning. The literature (ELL, AutoSkill/MUSE-Autoskill, SkillLearnBench,
   LifelongAgentBench) formalizes: experience exploration → long-term memory → abstracting
   recurring patterns into reusable skills → knowledge internalization.
2. **Multi-agent collaboration** — specialists + shared memory/plan; MCP (tools) + A2A
   (agent-to-agent delegation) as de-facto standards.
3. **Long-horizon autonomy** — the bar is moving toward multi-hour uninterrupted operation;
   computer-use stability via intent abstraction + shared plan memory.
4. **Security & governance as a hard gate (under-weighted)** — OWASP Top 10 for Agentic
   Applications; over-permissioning is the root cause of most incidents; indirect prompt
   injection and framework RCE are real; mitigations are tool allow-listing + **network
   egress control** + scoped/short-lived **agent identity** + per-operation authorization +
   tamper-evident audit.
5. **Digital-coworker framing + evals-first** — trajectory-level evaluation and boundary
   guardrails are standard in production.

**MicroClaw's strategic position — do not chase either reference project.** MicroClaw should
*not* take OpenClaw's heavy path (ChromaDB, enterprise SaaS dashboards) nor Hermes's path
(training clusters). Its defensible niche is the one neither occupies:

> A **single static Rust binary on a $5 VPS, channel-native, secure-by-default, that gets
> better the longer it runs** — no Python stack, no vector DB, no training cluster.

This collapses the roadmap into three strategic tracks (priority order):

- **Track A — Close the lifelong-learning loop (this document; the moat).** We already have
  skill self-review and graph memory; what's missing to truly close the ELL loop is (i)
  sleep-time offline consolidation (Pillar 1c), (ii) a skill curator that distills *and
  retires* skills from recurring trajectories (Pillar 2), and critically (iii) an
  **evaluation/regression gate** for skills and memory (Pillar 5) — without it, self-evolution
  has no quality brake and risks drift.
- **Track B — Promote security & governance to a first-class pillar (v0.4.0).** Best aligned
  with the "$5 VPS, self-hosted, secure-by-default" pitch. Align to OWASP Agentic Top 10:
  global **network egress control** (extend the existing `web_fetch` SSRF guard into a
  process-wide outbound policy, à la Proxyline), per-chat/per-agent **least-privilege tool
  authorization**, **tamper-evident audit** (hash-chain the existing audit log), and finish
  the sandbox backends (gVisor/SSH). Goal: the only small, self-hostable agent that is
  secure-by-default and audit-ready.
- **Track C — Long-horizon durability (v0.4.0 groundwork; "8-hour coworker").** Build on the
  scheduler/sub-agents/concurrency with durable long-task semantics, crash recovery, resumable
  runs, and first-class progress/ETA observability (closes the spirit of #307).

*Optional / low-priority (Hermes-adjacent, differentiator not main line):* an Atropos-compatible
trajectory **export schema** only — a researcher affordance, with no training-side investment.
Correctly deferred to 2027 in #378.

**Version cadence:** v0.3.0 = Track A + the eval gate (close the self-improving loop);
v0.4.0 = Track B (security/governance as a pillar) + Track C groundwork; the heavier #378
items (Canvas/A2UI, mobile nodes, serverless hibernate) stay further out.

## Pillar 1 — Self-curating memory (Letta-style consolidation)

MicroClaw's reflector runs *online* during the scheduler tick. Letta's 2026 design adds a
**sleep-time agent**: a background turn with no user message whose only job is to reorganize,
consolidate, and rewrite memory. Combined with the still-open hermes P0/P1 items in
`docs/hermes-agent-optimization-plan.md`, this becomes a tight, shippable workstream.

- **1a. PROFILE-priority injection + `USER.md`/`MEMORY.md` separation** (hermes P0, already
  scoped). Partition memories into PROFILE vs KNOWLEDGE/EVENT in `build_db_memory_context()`
  (`crates/microclaw-storage` / `memory_service.rs`); always inject the full PROFILE set
  first under a dedicated sub-budget so the agent always "knows the user" regardless of
  keyword relevance. Surface PROFILE as a distinct `USER.md` layer alongside the existing
  `AGENTS.md` file memory.
- **1b. Memory capacity management & auto-archive** (hermes P1). Add
  `memory_max_entries_per_chat` (default 200) and `memory_max_global_entries` (default 500);
  the reflector auto-archives lowest-confidence, least-recently-seen entries when over budget.
- **1c. Sleep-time consolidation pass** (Letta-inspired, *new*). A scheduler-driven
  background turn (reuse `scheduler.rs` + `process_with_agent(override_prompt=…)`) that, when
  a chat has been idle for N minutes, runs a no-user-message reflection turn whose mandate is:
  merge duplicate memories, promote recurring EVENTs into KNOWLEDGE, rewrite the PROFILE
  block, and prune stale graph triples. Off by default (`memory.sleep_time.enabled: false`),
  throttled like `idle_checkin`. This consumes the new auxiliary-model slot (Pillar 3) so it
  is cheap.
- **1d. Prompt-injection scan on memory content** (hermes P2). `scan_for_injection()` in a
  `memory_quality` path: reject invisible-unicode / instruction-override / exfiltration
  patterns before a memory is stored. (Pairs with the input guardrail in Pillar 5.)

**Exit criteria:** PROFILE always injected; per-chat memory bounded; an opt-in idle
consolidation pass measurably reduces duplicate/stale memories on a fixture chat; injected
memory content is scanned.

## Pillar 2 — Skill Curator (self-distilling skills)

This is the headline #378 Q3 item and the clearest "self-improving" feature. Today the agent
*uses* 41 built-in skills and can `skill_manage` them by hand; it does not yet *create* skills
from its own experience.

- **2a. Background curator loop** in `crates/microclaw-app/src/skill_curator.rs`, riding the
  existing `scheduler.rs` cadence. Periodically scan recent session trajectories
  (`session_search` / `messages`), detect repeated multi-step patterns (≥5 tool calls
  recurring across sessions), and propose a `skill_manage create/patch`.
- **2b. Post-task review** (hermes P3): after a complex task (≥5 tool calls) optionally fire a
  lightweight auxiliary-model call to decide whether a reusable skill is warranted; write a
  per-run report under `docs/reports/` style output.
- **2c. Safety**: every generated skill goes through the existing ClawHub
  `gate.rs`/security-scan path, is created **disabled**, and creation is restricted to control
  chats. Archive stale/never-activated curated skills automatically.
- **2d. Progressive-disclosure hygiene** (Claude Agent SDK best practice): curator-authored
  `SKILL.md` files must keep a thin top-level description and push detail into linked files so
  they load on demand rather than bloating the catalog.

**Exit criteria:** on a seeded set of repetitive sessions, the curator proposes at least one
correct, security-scanned, disabled-by-default skill, with a report; no curator output can run
without explicit activation.

## Pillar 3 — Per-task auxiliary models

Straight from hermes parity in #378, and a prerequisite that makes Pillars 1c and 2 cheap.
Today one model pays for everything including ancillary work.

- Add optional model slots: `aux_models.compaction`, `aux_models.title`,
  `aux_models.vision_describe`, `aux_models.session_search_summarize`,
  `aux_models.memory` (each falls back to the main model when unset).
- Wire them through `runtime.rs` provider initialization so the auxiliary slots reuse the
  existing provider-agnostic `LlmProvider` factory — no new provider code, just additional
  configured instances.
- Route the relevant call sites: compaction in `agent_engine.rs`, title generation, the
  `describe_image` tool, `session_search` result summarization, and the reflector/curator.
- Surface per-slot token usage in the existing usage/observability views so operators can see
  the cost split.

**Exit criteria:** with an aux model configured, compaction/title/vision calls demonstrably
hit the cheaper model while the main turn stays on the primary; `/usage` shows the split.

## Pillar 4 — Sandbox backend pluralization & defense-in-depth

The 2026 sandbox consensus (microVM / gVisor / hardened container) is the strongest external
signal, and #378 lists "sandbox backend pluralization" as a Q3 parity item. MicroClaw already
has the `microclaw-tools` sandbox abstraction and the `path_guard` module; this pillar makes
`risk = exclusive` tools (notably `bash`) run in a real isolation boundary.

- **4a. Backend trait + selection.** Extend the sandbox abstraction with pluggable backends:
  `none` (current default), `docker`, `gvisor` (runsc runtime), and `ssh` (remote host).
  Selected via `sandbox.backend`; nanoclaw's per-group container model is the reference for
  the Docker backend (per-chat container, explicit mounts only, no host FS by default).
- **4b. Defense-in-depth defaults.** For the Docker/gVisor backends: drop capabilities,
  seccomp/AppArmor profile, no-network-by-default with an allowlist, read-only root + scoped
  writable mount, and resource limits (CPU/mem/pids/time). Keep `path_guard` as the
  application-layer second line.
- **4c. Credential hygiene** (nanoclaw lesson): the sandboxed process never receives raw API
  keys; secrets stay in the host process and are injected only at the egress boundary.
- **4d. Graceful degradation.** If the configured backend isn't available at startup,
  `microclaw doctor` reports it and the runtime refuses `exclusive` tools rather than silently
  falling back to host execution.

**Exit criteria:** `bash` runs inside a Docker (and, where available, gVisor) sandbox with
no-network + scoped-mount defaults; a path-escape and a network-exfil attempt are both blocked
in tests; `doctor` validates backend availability.

## Pillar 5 — Trajectory evals & boundary guardrails

The 2026 production pattern is (a) evaluate full *trajectories* — tool choice, argument
validity, step count, termination, final-answer quality — not just final answers, and (b)
place guardrails at every model boundary (pre-input, post-retrieval, pre-tool-call,
post-output). MicroClaw has regression packs and OTLP traces but no replayable eval harness or
boundary guardrail layer.

- **5a. Trajectory eval harness.** A `microclaw eval` subcommand that replays recorded
  sessions (the full `Vec<Message>` including tool_use/tool_result is already persisted) as
  fixtures and scores: did it pick the right tool, were arguments valid, did the loop
  terminate within budget, did the final answer satisfy a rubric. Output JSON for CI gating.
  Build on the existing `docs/test`/regression-pack infrastructure.
- **5b. Boundary guardrails.** A small guardrail layer in `agent_engine.rs` with hooks at
  pre-input (combine with 1d memory-injection scan), pre-tool-call (argument/policy checks,
  reuse `risk` levels), and post-output (PII/secret leak scan before delivery). Off-by-default
  policies, configurable, logged to the audit trail.
- **5c. Iteration-budget progressive warnings** (hermes P1, small but high-value): inject a
  "start wrapping up" hint at 70% of `max_tool_iterations` and an "URGENT: final answer now"
  at 90% so the loop ends gracefully instead of abruptly.
- **5d. Optional RL/trace export groundwork** (#378 2027 item, started early): make the
  trajectory recorder emit an Atropos-compatible JSONL so the eval fixtures double as
  fine-tuning data. Schema only in 0.3.0; full export later.

**Exit criteria:** `microclaw eval` runs a fixture suite in CI and fails on regressions;
pre-tool-call and post-output guardrails are wired and tested; iteration warnings fire.

## Release shape

| Order | Workstream | Depends on | Risk | Default |
|---|---|---|---|---|
| 1 | P3 aux models | — | low | opt-in (falls back to main) |
| 2 | P1a/b/d memory profile + caps + scan | — | low | on (PROFILE), bounded caps |
| 3 | P5c iteration warnings | — | low | on |
| 4 | P1c sleep-time consolidation | P3 | medium | **off** |
| 5 | P2 skill curator | P3 | medium | **off** |
| 6 | P4 sandbox backends | — | high | `none` (current behavior) |
| 7 | P5a/b eval harness + guardrails | P1d | medium | eval in CI; guardrails opt-in |
| 8 | P5d RL export schema | P5a | low | off |

Sequencing rationale: ship the cheap, safe, on-by-default wins first (aux models, memory
profile, iteration warnings); gate the autonomous/self-improving features (sleep-time,
curator) behind flags and behind the aux-model slot so they're cheap; treat sandbox
pluralization as its own high-risk track that never changes default behavior; land evals last
so they can assert the rest didn't regress.

### Compatibility & safety guarantees

- **No default behavior change** for existing deployments: every new autonomous or
  isolation-changing feature is **off by default**; aux-model slots fall back to the main
  model when unset.
- All new config keys documented in `microclaw.config.example.yaml` and
  `docs/generated/config-defaults.md`; `node scripts/generate_docs_artifacts.mjs --check` must
  pass.
- Schema migrations are additive and versioned (curated-skill provenance, eval runs).
- Per-PR checklist from `docs/roadmap/feature-completion-tracking-board.md` applies:
  `cargo test -q`, web build, docs artifacts check, rollback note.

### Explicitly deferred to v0.4.0+ (from #378 Q4 / 2027)

Live Canvas / A2UI (agent-driven web UI), workspace multi-agent routing, voice wake / menu-bar
companion, mobile node clients, serverless hibernate backends (Modal/Daytona/Vercel), full RL
trajectory export, Termux target, OpenClaw migration tool, ClawHub ratings/signatures. These
are real and tracked in #378 — they're just a release beyond this self-improving-runtime core.

## Open decisions (need a maintainer call)

1. **Sandbox default in 0.3.0** — keep `none` (zero behavior change, isolation strictly
   opt-in) or make `docker` the recommended default in `setup`/docs while keeping `none` the
   code default? Recommendation: keep code default `none`, make `docker` the *recommended*
   path in the wizard.
2. **Curator vs USER.md ordering** (#378 decision point 3) — the curator benefits from reading
   PROFILE context, so land Pillar 1a before Pillar 2. This plan already sequences it that way.
3. **gVisor scope** — ship gVisor as a first-class backend in 0.3.0, or Docker + SSH first and
   gVisor as a fast-follow? Recommendation: Docker + SSH in 0.3.0, gVisor documented + behind
   a feature check.
